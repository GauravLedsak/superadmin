# Security & Access — [Mock]
Route: `security` · Source: `superadmin/src/pages/SecurityPage.jsx` + `superadmin/src/data/security.js`

## Purpose
Manages LEDSAK's **internal admin-team pool** (the staff who operate this Super Admin console) — distinct from `UsersPage`, which manages tenant end-users of the CRM product. Eight tabs: Admin Users, Roles & Permissions, Sessions, IP Restrictions, Login History, Security Alerts, API Keys, Webhook Keys. Replaces the previous 3-tab hardcoded stub (Roles/Sessions/Policies, all `store.notify()`-only with no state changes).

## Flow

### Primary path
1. Page loads on the **Admin Users** tab. Four KPI cards (Admin Users, Privileged Admins, Active Sessions, Security Alerts (7d)) compute live from store state.
2. "Audit Roles" button in the header jumps to the Roles & Permissions tab. "+ New Admin" opens the create-admin drawer from any tab.
3. Each tab has its own filter bar, table, and row actions; mutations go through `StoreContext` methods that call `addHistory(...)` (audit trail) and `notify(...)` (toast), matching the app-wide convention.

There is no MFA in this app — admin auth is single-factor (password only, `resetAdminPassword` sends a reset link). An earlier draft of this page included MFA enrollment/toggle/policy fields; these were removed as a feature LEDSAK doesn't have. Login events and security alerts model brute-force/lockout scenarios but nothing MFA-specific.

### Tab-by-tab
- **Admin Users**: search + role/status filters. Row menu: Edit, Reset Password, Suspend/Reactivate, Deactivate. Impersonate icon reuses `store.impersonate`. Create/Edit share one drawer (`AdminFormDrawer`).
- **Roles & Permissions**: list view (role, assigned-admin count, a permissions summary string, type badge) → clicking a row or "Edit Permissions" opens the **permission matrix editor**: one Card per `PERMISSION_SECTIONS` group, each rendering a grid of modules × actions. The `view` action is a scope `<select>` (All / Assigned Only); every other action is a checkbox; actions a module doesn't support render "—". Edits are held in local state until "Save Permissions" commits them via `store.updateRolePermissions`. "+ New Role" → name/description/type modal → creates an all-empty-permissions role and opens straight into its editor. Duplicate/Delete via row menu (delete blocked if the role is a system role or has assigned admins).
- **Sessions**: a collapsible "Security Policy" panel (max failed attempts / lockout duration / session timeout, wired to `store.updateSessionPolicy`) sits above the session table. Per-row Revoke (disabled + tooltipped for the current session); "Revoke All" prompts a confirm modal.
- **IP Restrictions**: separate Allowlist/Blocklist cards, each its own table. "+ Add Rule" validates IPv4/IPv6/CIDR format and warns if a new Block entry duplicates an existing Allow entry. Status is a `Switch` toggle (Active/Disabled); Remove is immediate (no confirm — low-consequence, reversible by re-adding).
- **Login History**: read-only. Reuses `applyLogFilters`/`sortRows`/`exportLogs`/`DATE_PRESETS` from `data/logs.js` rather than reimplementing filtering — this data is shaped like a log stream even though it lives in the Security module. Rows with `account.locked` or 3+ same-IP `login.failed` entries get a `dangerSoft` row background and a "⚠ Repeated failures" badge.
- **Security Alerts**: status workflow Open → Investigating → Resolved/False Positive. Row menu offers quick actions (Resolved/False Positive use `window.prompt` for the required note — a pragmatic shortcut versus the detail drawer, which has a proper inline note field via `AlertNotePrompt`). Notes accumulate as a timeline, never overwritten.
- **API Keys**: create with scope checkboxes (`full_access` is exclusive — selecting it clears/disables the others). On create or rotate, a one-time `KeyRevealModal` shows the generated key value (`lsk_{env}_{24 hex chars}`, client-side dummy — never persisted). Rotate = revoke old + create new with the same name/scopes.
- **Webhook Keys**: per-tenant outbound webhook subscriptions — distinct from API Keys (inbound, platform-wide REST access). Creation is a 3-step wizard matching the requested flow: **Step 1** select the client (from `store.clients`, the same tenant list as Clients/Tenant 360 — no separate tenant list is maintained here); **Step 2** select one or more modules (`WEBHOOK_MODULES` in `data/security.js`: Leads, Users, Subscriptions, Payments, Onboarding, Automation); **Step 3** multi-select events, grouped by the modules chosen in Step 2 (unchecking a module in Step 2 drops its events from the Step-3 selection), plus the webhook name and destination URL. On create, a one-time reveal modal shows the signing secret (`whsec_{32 hex chars}`, dummy, never persisted) — same one-time-reveal pattern as API Keys. Row menu: View Details, Regenerate Secret (issues both a new key ID and a new secret, both shown once via the reveal modal), Revoke.

The module/event vocabulary here (`lead.created`, `payment.failed`, `subscription.renewed`, etc.) intentionally matches the event types already seeded in the Logs & Audit page's Webhook Logs tab (`data/logs.js`), so a key configured here and a delivery attempt logged there describe the same events — though the two are not currently wired together (creating a key here doesn't generate corresponding log rows).

## Rules
- **Two user pools, not one**: `adminUsers` (this page) vs. `users` (CRM end-users, `UsersPage`) are separate store arrays — nothing here reads or writes `store.users`.
- **Permission schema**: `PERMISSION_SECTIONS` in `data/security.js` defines 8 sections × ~2-4 modules each, each module listing which of ~15 possible actions it supports. `view` is scope-based (`SCOPED_ACTIONS`); everything else is boolean. `summarizePermissions()` derives the list-view summary string live from a role's `permissions` object — nothing is duplicated/cached.
- **Deletion guards**: a role can't be deleted if `isSystem` is true or any admin is currently assigned to it (checked both in the row-menu handler and again in `store.deleteRole` itself — belt and suspenders, since the menu check alone wouldn't stop a stale-closure or a second caller).
- **Immutability of history**: `loginHistory` has no setter exposed from the store — it's seeded once and never mutated, consistent with it being an audit trail.
- All mutations follow `setState → addHistory({...}) → notify(...)`, matching the exact pattern already used by plan/subscription/tenant methods in `StoreContext.jsx`.

## Permissions
This page is itself what would gate the rest of the app's permissions in a real system, so it has no *access* gate of its own beyond being a normal nav route — every super admin viewing this demo can reach and use every tab. (Contrast with Logs & Audit's page-local "Viewing as" role simulator — Security & Access doesn't have an equivalent self-gating switcher; see Open questions.)

## Automations
None triggered from this page. `SEED_SECURITY_ALERTS` is static seed data representing *hypothetical* past detections (brute-force, suspicious login, role escalation, API-key-abuse, session-anomaly) — there's no live detection engine generating new alerts as admin actions occur elsewhere in the app.

## Decisions
- 2026-07-23 · Store state (not a self-contained page module like `data/logs.js`) · Rejected: a `data/security.js`-owned localStorage module. Reason: the prompt's own spec required `StoreContext.jsx` extension with `addHistory`/`notify` on every mutation — admin/role identity is core-enough state that other pages could plausibly need it later (e.g. showing "last edited by" using a real admin list), unlike Logs' high-volume, page-scoped records.
- 2026-07-23 · Login History reuses `data/logs.js` filter/sort/export helpers rather than duplicating them · Rejected: copy-pasting `applyLogFilters`/`sortRows`/`exportLogs` into `data/security.js`. Reason: login-history rows are structurally a log stream (timestamp, event, outcome) even though they're not one of the Logs page's 7 types; reuse avoids two divergent filtering implementations.
- 2026-07-23 · Alert quick-resolve from the row menu uses `window.prompt` instead of an inline form · Rejected: building a second inline-note UI for the menu path. Reason: the detail drawer already has the spec'd inline note field (`AlertNotePrompt`); the menu shortcut is for a fast one-off resolution and a native prompt is a reasonable trade for not duplicating that component in a `<Menu>` item, which can't easily host expanding form state.
- 2026-07-23 · Session policy (`SEED_SESSION_POLICY`) surfaced as a collapsible panel inside the Sessions tab, not its own tab · Rejected: an 8th tab. Reason: the spec's tab list is fixed at 7 and doesn't include one for global auth policy, but a working policy-update flow was still required — Sessions was the closest existing tab (`sessionTimeoutMinutes` already displayed there).
- 2026-07-23 · Removed MFA entirely (enrollment fields, MFA badge/filter/toggle, `mfa_disabled` alert type, `mfa.challenge`/`mfa.verified` login events, MFA policy toggles) · Rejected: keeping it as a mocked/decorative feature. Reason: LEDSAK doesn't have an MFA feature — the app should only describe/mock capabilities the product actually has or is planning, not invent security features that don't exist. Auth is single-factor (password + reset link) throughout.
- 2026-07-23 · Webhook Keys added as an 8th tab, reusing `store.clients` for the client picker rather than a new tenant list · Rejected: duplicating tenant data into `data/security.js`. Reason: `store.clients` (from `data/seed.js`, backing Clients/Tenant 360) is already the single source of truth for tenants app-wide; a webhook key is fundamentally "this tenant, these events," so it should reference the same client records everything else does.
- 2026-07-23 · Webhook module/event vocabulary (`WEBHOOK_MODULES`) deliberately mirrors the event names already used in `data/logs.js`'s Webhook Logs seed (`lead.created`, `payment.failed`, etc.) · Rejected: an independent event taxonomy. Reason: these two features describe the same underlying concept (outbound webhooks) from two angles — subscription config here, delivery history there — and should speak the same event vocabulary even though the two aren't yet wired together (see Open questions).

## Open questions
- No role-based gate exists on this page itself — should viewing/editing Security & Access require the `security` module permission it defines (`manageAdmins`, `manageRoles`, etc.) once real auth exists? Right now those permission keys are purely descriptive, not enforced anywhere in the UI.
- `Security Alerts` are static seed data, not generated from actual admin actions taken elsewhere in the app — should mutating admin/role state trigger a synthetic alert for demo realism?
- API key values are dummy hex strings generated client-side and never stored (even in memory beyond the reveal modal) — confirm this is intentional for the prototype rather than a gap to fill with a real generation/hashing flow. Same applies to Webhook Key signing secrets (`whsec_...`).
- `window.prompt` for row-menu Resolve/False-Positive is a native browser dialog, visually inconsistent with the rest of the app's UI — worth replacing with a small inline popover if this page graduates beyond prototype.
- Webhook Keys created here don't produce corresponding rows in the Logs & Audit page's Webhook Logs tab (`data/logs.js`) — creating a key doesn't simulate any deliveries. Should creating/using a webhook key seed a delivery log entry for demo realism, or should these stay independent (config vs. history)?
